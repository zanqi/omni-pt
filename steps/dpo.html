DPO experiment: correcting babble-track over-triggering

Qwen3-Omni-30B-A3B-Instruct, babble track only, continuing from the existing SFT adapter.

Decisions locked in for this run
  • Rejected samples: on-policy, mined from the current SFT model's own rollouts — no synthetic/templated negatives.
  • Model: Qwen3-Omni-30B-A3B-Instruct + its existing Qwen3-Omni-30B-A3B-Instruct-bab-sft adapter (proving the approach on the larger model first, not Qwen2.5-Omni-3B).
  • Track: babble only (background-noise track) — where the over-triggering was actually observed.
  • Trainer: custom minimal DPO loop, not TRL — TRL isn't installed in qwen3omni/qwen25omni, and this repo's Qwen3OmniMoeForConditionalGeneration thinker-only wrapper + hand-built audio collator (sft_qwen.py) has no guaranteed DPOTrainer compatibility.
  • Base checkpoint: continue from the SFT adapter, not a fresh LoRA from base — we're correcting a behavior that adapter already learned.

Why this is needed

Pulled straight from bab_results_Qwen3-Omni-30B-A3B-Instruct-bab-sft_slurp.jsonl (150-row test-split eval, already run):

answer_rows: 50 — 18 over-triggered (36%): the model asked for clarification/repeat on audio that was fully intelligible. 
kind: "answer"   sentence: "event reminder mona tuesday"
response: "Did you mean Tuesday or another day?"
judged_type: "repair"   score: 0.0
Confusion matrix from that run: answer→answer: 28, answer→repair: 14, answer→repeat: 4, answer→bad: 4.

The scoring rule already in babble_eval_qwen.py (SCORE_MATRIX, keyed by [row kind][judged_type]) is exactly the reward signal this experiment needs — it already encodes both failure directions, not just over-triggering:

row kindjudged_typescoremeaning
answeranswer1.0correct, no negative signal
answerrepair / repeat0.0over-triggering — the bug you found
answerbad0.0hallucinated on clean audio
repairanswer1.0correctly resolved the ambiguity — not a guess, since classify_response's rubric only allows "answer" when every detail is actually correct
repairrepeat0.5over-scoped repair (asked to repeat everything when only one detail was lost)
repairbad0.0hallucination on ambiguous audio — the more dangerous mirror-image failure, worth including
repeatrepair0.5under-scoped (single-detail question when the whole utterance was lost)
repeatanswer / bad0.0–1.0*see kind=answer row above for the "answer" case (1.0, genuinely resolved); "bad" is 0.0

*Note repeat→answer is also 1.0 by this same "only correct answers get classified answer" logic. The only genuinely bad cells are *→bad and answer→{repair,repeat}.

This means every rollout scoring <1.0 is a ready-made rejected sample, and the gold target text already in the dataset is the chosen sample — no new judging logic needs to be written, and no synthetic negatives are needed.

Pipeline

0Prerequisite: don't clobber the existing eval file

babble_eval_qwen.py's output path is bab_results_<model_or_adapter_name>_slurp.jsonl — derived only from the model/adapter name, not the split. Running it with --split train would silently overwrite the real test-split eval result you already have for this adapter. Add a one-line --out override before running anything:

ap.add_argument("--out", default=None, help="Defaults to bab_results_<name>_slurp.jsonl")
...
out_path = args.out or f"bab_results_{model_name}_slurp.jsonl"

1Harvest on-policy rollouts over the train split

No new generation code needed — babble_eval_qwen.py's main loop already does exactly this per row (generate → classify_response → SCORE_MATRIX lookup → write JSON record with response/judged_type/score). Just point it at the train split with the --out override:

conda activate qwen3omni
JUDGE_HOST=$(cat /gscratch/sciencehub/zanqil/vllm_judge/vllm_judge_host.txt)
python babble_eval_qwen.py \
    --dataset keylazy/slurp-babble-Qwen3-Omni-30B-A3B-Instruct-v1 \
    --split train --num-rows -1 \
    --model-path Qwen/Qwen3-Omni-30B-A3B-Instruct \
    --adapter-path ./Qwen3-Omni-30B-A3B-Instruct-bab-sft \
    --judge-base-url "http://${JUDGE_HOST}:8000/v1" \
    --out bab_train_rollouts_Qwen3-Omni-30B-A3B-Instruct-bab-sft.jsonl

3000 train rows × one generation + one judge call each. Extrapolating the test-split rates above (~36% non-1.0 on answer rows, ~30% on repair, ~26% on repeat) suggests roughly 900–1000 usable pairs out of 3000 rows — log the actual yield and don't top it up with synthetic data if it comes in lower; that's a real signal about how often the SFT model still misbehaves.

2Build the preference-pair dataset — new script babble_dpo_data.py

  • Read bab_train_rollouts_..._.jsonl, keep rows with score < 1.0.
  • Join back to keylazy/slurp-babble-Qwen3-Omni-30B-A3B-Instruct-v1 (train split) by id to reattach the audio column (the eval script only logs text fields, not raw audio arrays).
  • Emit one row per pair: {id, kind, audio, sentence, chosen: target, rejected: response, rejected_type: judged_type, score}.
  • Carve a small held-out slice (~10%) for tracking DPO loss/margin on unseen pairs during training — the real evaluation is still step 4's full babble_eval_qwen.py run on the untouched test split.
  • Push to keylazy/slurp-babble-dpo-Qwen3-Omni-30B-A3B-Instruct-v1.
  • Log the breakdown by (kind, rejected_type) so the class balance is visible before training — e.g. if it's 90% answer→repair pairs and almost no repair→bad/repeat→bad pairs, DPO will mostly learn "stop over-asking" and barely touch the hallucination-on-ambiguous-audio failure mode.

3Custom DPO training loop — new script dpo_qwen.py

Reuses sft_qwen.py's load_processor, get_sft_model_cls("qwen3"), find_lm_linear_names, and the full/prompt-diff labeling trick from OmniSFTCollator — just applied to two assistant completions (chosen, rejected) sharing the same audio+prompt prefix instead of one.

Reference model without doubling GPU memory

Loading a second full copy of a 30B model for the DPO reference is wasteful when the reference is the exact SFT adapter you already have. Instead, attach it twice to one frozen base model as two named PEFT adapters:

model = PeftModel.from_pretrained(base, sft_adapter_path, adapter_name="policy", is_trainable=True)
model.load_adapter(sft_adapter_path, adapter_name="reference", is_trainable=False)

# per step:
model.set_adapter("reference")
with torch.no_grad():
    ref_logp_chosen, ref_logp_rejected = seq_logprobs(model, batch)

model.set_adapter("policy")
policy_logp_chosen, policy_logp_rejected = seq_logprobs(model, batch)   # requires grad

loss = dpo_loss(policy_logp_chosen, policy_logp_rejected,
                 ref_logp_chosen,   ref_logp_rejected,  beta=0.1)
loss.backward()

seq_logprobs = sum of per-token log-probs over the completion span only (same labels != -100 mask OmniSFTCollator already builds, just computed for both the chosen and rejected assistant turns).

DPO loss (Rafailov et al. 2024):

logits = beta * ((policy_logp_chosen - ref_logp_chosen)
                - (policy_logp_rejected - ref_logp_rejected))
loss = -F.logsigmoid(logits).mean()

Memory

Single A100 80GB is available on this node. bf16 Qwen3-Omni-30B-A3B weights are ~60GB just for storage (MoE reduces active compute, not weight footprint) — already tight before DPO's extra forward passes. Recommend:

  • --qlora (4-bit NF4 base, already implemented in sft_qwen.py's load_model) to free headroom for the 4 forward passes (chosen/rejected × policy/reference) per step.
  • Gradient checkpointing (already used in sft_qwen.py), micro-batch size 1, gradient accumulation to reach an effective batch size of 8–16.
  • If it still OOMs, request 2 GPUs and shard the frozen base with device_map="auto" instead of the hardcoded {"": 0} in load_model.

Smoke test (mirror sft_qwen.py --smoke)

At initialization, policy and reference are identical adapters, so policy_logp == ref_logp for both chosen and rejected, which forces logits == 0 and loss == -log(0.5) ≈ 0.693 exactly. Assert this before launching a real run — it's a strong, free correctness check on the dual-adapter plumbing.

Hyperparameters to start with

paramstarting valuenote
beta0.1lower = larger policy shift allowed per step; sweep 0.05–0.3 if EAR barely moves or moves too fast

4Evaluate against the SFT baseline

No new eval code — rerun the existing script, pointed at the saved DPO adapter, same test split used for the baseline number (EAR 0.705, C=0.56 driven down by the 36% over-trigger rate):

python babble_eval_qwen.py \
    --dataset keylazy/slurp-babble-Qwen3-Omni-30B-A3B-Instruct-v1 --split test \
    --model-path Qwen/Qwen3-Omni-30B-A3B-Instruct \
    --adapter-path ./Qwen3-Omni-30B-A3B-Instruct-bab-dpo \
    --judge-base-url "http://${JUDGE_HOST}:8000/v1"

Compare confusion matrices cell-by-cell against the baseline, not just aggregate EAR:

  • Must improve: answer→repair + answer→repeat (currently 14+4=18/50) should shrink — this is the whole point.
  • Must not regress: repair→answer (8/50) and repeat→answer (2/50) should not grow. If DPO overcorrects, the model swings from "over-asks" to "confidently hallucinates on ambiguous audio," which is worse than the bug it fixes — that failure mode is scored 1.0 by SCORE_MATRIX only when the guess happens to be correct; a wrong guess is bad (0.0), so watch repair→bad / repeat→bad too.
Risk: reward hacking on response shape, not content. DPO can learn shallow patterns that flip judged_type without real improvement (e.g. always tacking on "...right?" to sound more like "answer" per the rubric's phrasing rules). Manually read 15–20 sample outputs across all three kinds after training, not just the aggregate score.
Risk: pair-count imbalance skews what gets fixed. If step 2's yield is dominated by answer→{repair,repeat} pairs (very likely, since that's the majority failure mode measured), DPO will preferentially learn "stop over-asking" and may leave the rarer *→bad hallucination pairs under-corrected. If that matters more than the over-triggering fix, consider oversampling the bad-labeled pairs when building the training set in step 2.

Open questions before running step 2/3 for real

  • Should the harvest pass (step 1) use a single greedy rollout per row (matches how the model is evaluated/used, simplest), or a few sampled rollouts per row (do_sample=True, temperature > 0) to increase negative yield on rows the greedy decode already gets right? Starting with greedy-only is the simpler default and directly reflects real deployment behavior.
  • Confirm whether the existing Qwen3-Omni-30B-A3B-Instruct-bab-sft adapter was trained with --qlora or full bf16 — the DPO reference and policy adapters must be loaded onto a base model with matching precision to the checkpoint they were trained against, or the "identical logp at init" smoke-test assertion will fail for a spurious reason.
